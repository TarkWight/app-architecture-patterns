# Roadmap

Документ фиксирует ближайшую дорожную карту развития MVP ПО стенда.
Шаги перечислены так, чтобы их можно было закрывать отдельными git-коммитами.

## 1. Отчет PDF по старому макету

Цель: повторить внешний вид и структуру отчета из `legacy`, но на текущем C++/Qt стеке.

Что сделать:

- расширить `PdfDocumentModel` понятиями отчета: организация, лицензия, адрес, тип испытаний, дата, оператор, комментарий, заключение;
- добавить таблицу параметров испытаний: тип, формула силы ветра, расчетное время, время теста, годен/негоден;
- перерисовать отчет в `QtPdfExporter` через `QPdfWriter` и `QPainter`;
- использовать текущий `PlotRenderer` для графиков, без внешних PDF/plot библиотек;
- сохранить подписи графиков из legacy-отчета, но привести термины к документации проекта.

Критерии готовности:

- PDF содержит титульную часть, таблицу, график управления, график телеметрии, комментарий, заключение и ФИО оператора;
- отчет генерируется из текущего состояния приложения;
- сборка проходит с clang-format и clang-tidy.

Решение MVP:

- верстку отчета пока держать в коде `QtPdfExporter`;
- TOML использовать позже для текстовых шаблонов, но не для координат и layout-логики.

## 2. Базовый движок формул

Цель: заменить простой stub на минимально полезную обработку математических выражений.

Что сделать:

- определить доменную модель `ControlFormula`;
- поддержать переменную времени `t`;
- поддержать базовые операции и функции: `+`, `-`, `*`, `/`, скобки, `sin`, `cos`;
- возвращать явную ошибку валидации формулы, а не молча строить пустой график;
- подключить формулу к построению графика управления.

Критерии готовности:

- формулы вида `0`, `1.5*t+7`, `0.4*sin(0.5*t)*sin(t)` рассчитываются;
- ошибка формулы видна оператору через предупреждение и лог;
- `BuildControlPlotUseCase` не зависит от деталей парсинга.

## 3. Движок сценариев испытаний

Цель: подготовить режим заготовленного и гибридного испытания.

Что сделать:

- ввести `ControlScenario` как набор воздействий/шагов во времени;
- описать шаг сценария через целевое воздействие, длительность и правило перехода;
- связать сценарий с типом испытания;
- добавить режимы выполнения: ручной, заготовленный, гибридный;
- для гибридного режима сохранить правило: оператор может задать воздействие, после чего система плавно возвращается к сценарию.

Критерии готовности:

- ручной режим остается текущим рабочим режимом;
- заготовленный режим блокирует ручное управление;
- гибридный режим имеет явную заглушку или минимальную реализацию без скрытого поведения;
- сценарий можно выполнить без обращения к UI-логике.

## 4. DDD-модель и инварианты

Цель: убрать анемичность из ключевых моделей и перенести правила предметной области из UI/use cases в домен.

Что сделать:

- заменить свободные `double` для воздействия на value objects:
  - `BeaufortForce`, допустимый диапазон `0..7`;
  - `WindDirection`, нормализация `0..359`;
  - `AngleOfAttack`, допустимый диапазон стенда;
- выделить `StandImpact` как целевое управляющее воздействие;
- вынести плавный переход воздействия из `MainWindow` в доменную модель или доменный сервис;
- оформить `TestSession` с явными переходами состояний испытания;
- оформить `StandConnection` или `StandConnectionLifecycle` с явными состояниями связи;
- оформить `TelemetryCurve` с источником данных, цветом, видимостью и привязкой к оси графика;
- оформить `TestProtocol` как объект, который проверяет полноту данных для отчета.

Критерии готовности:

- UI не владеет правилами диапазонов и переходов;
- use cases вызывают поведение доменных объектов, а не вручную меняют набор полей;
- невалидные значения возвращают ошибку/результат валидации;
- существующие use cases остаются покрыты текущими сценариями из `UC.md`.

Заметка после MVP:

- диапазон Бофорта `0..7` оставлен по поведению legacy-приложения; нужно выяснить реальные ограничения стенда и модели
  ветровой нагрузки;
- диапазон угла атаки `0..360` оставлен по поведению legacy-приложения; нужно проверить реальные механические и
  протокольные ограничения стенда.

## 5. Очистка legacy counter

Цель: убрать остатки демонстрационного счетчика, который не относится к предметной области стенда.

Что сделать:

- удалить `SwitchCounterUseCase`;
- удалить `ICounterRepository`;
- удалить `CounterSnapshot`;
- удалить неиспользуемые include и зависимости из CMake;
- проверить упоминания `Counter`, `CounterDemo`, `counter` в `src`.

Критерии готовности:

- проект собирается без counter-артефактов;
- в `src` не остается use case или port, не связанный со стендом, испытаниями, графиками, отчетом или телеметрией.

## 6. Согласование use cases с кодом

Цель: использовать `UC.md` как карту сценариев, а не как отдельный документ.

Что сделать:

- сопоставить каждый пункт `UC.md` с текущими use cases;
- переименовать сценарий `Скрыть кривую графика оси` в `Показать/скрыть кривую` или `Управлять видимостью кривой`;
- разделить сценарии просмотра, настройки и выполнения испытания;
- для отсутствующих сценариев создать application use case или явно отметить их как будущие.

Критерии готовности:

- каждый пользовательский сценарий имеет понятное место в application layer;
- UI-команды не обходят application/use case слой;
- расхождения между `UC.md` и кодом явно отмечены.

## 7. Диагностический режим и расширение графиков

Цель: заранее разделить операторские графики, управляющие команды и диагностические данные стенда, чтобы позднее
добавить режим анализа без переписывания основного сценария испытаний.

Что учитывать:

- график телеметрии отображает то, что стенд вернул;
- график управления отображает то, что задано оператором или рассчитано формулой, и то, что система реально отправляет
  на стенд после ограничений;
- служебные параметры стенда не выводятся в основной MVP-интерфейс без явной необходимости;
- диагностический режим или отдельная вкладка могут появиться позднее.

Потенциальные диагностические данные:

- напряжение питания, уже присутствующее в телеметрии;
- температура приводов, если будет подтвержден способ ее получения;
- дополнительные параметры приводов, если они будут доступны через отдельный IP/API или другой канал обмена.

Линии графика управления:

- `target_value` — значение, рассчитанное по формуле или заданное оператором;
- `operator_input_value` — значение, введенное оператором вручную;
- `safe_command_value` — значение, фактически отправляемое на стенд после ограничений, сглаживания и проверок
  безопасности;
- `stand_feedback_value` — фактическое значение стенда, если оно относится к анализируемому воздействию и приходит в
  телеметрии.

Поведение по режимам:

- автоматический режим: формула полностью определяет управляющее воздействие; на графике достаточно показывать
  рассчитанную функцию и, при необходимости, фактически отправленное значение;
- гибридный режим: формула задает базовое значение, оператор может вносить отклонения; на графике нужно различать
  значение формулы, значение с учетом операторского отклонения и безопасную команду;
- ручной режим: оператор напрямую задает параметры управления; на графике нужно различать введенную команду и безопасную
  траекторию, реально отправленную на стенд.

Архитектурное следствие:

- не смешивать целевое значение, пользовательский ввод, безопасную команду и телеметрию стенда в одну модель;
- проектировать `PlotModel`/серии графика так, чтобы в будущем можно было добавить новые линии без смены сценариев;
- диагностические данные должны подключаться как отдельное представление или режим, а не как обязательная часть MVP.

## 8. Проверка безопасности UI-состояний и многопоточности

Цель: убедиться, что обновление интерфейса, сетевые callbacks и таймеры не создают циклические сигналы, re-entrant
обновления состояния и обращения к UI не из GUI-потока.

Что проверить:

- места, где `SessionStateQtAdapter` эмитит сигналы, а UI в ответ меняет widgets, способные снова вызвать application
  use case;
- использование `QSignalBlocker` или явных guard-флагов при программном обновлении `QComboBox`, `QSpinBox`,
  `QDoubleSpinBox`, `QLineEdit` и checkbox-состояний;
- отсутствие циклов вида `state changed -> set widget value -> widget signal -> use case -> state changed`;
- поток, из которого вызываются callbacks телеметрии, статусов соединения и ошибок;
- гарантии, что все изменения Qt widgets выполняются в GUI thread;
- безопасность `SessionState` при вызовах из сетевых callbacks, таймеров испытания и UI-команд;
- корректность остановки timers/polling при `Stop`, disconnect и закрытии приложения.

Критерии готовности:

- программные обновления UI не вызывают повторные команды в application layer без действия оператора;
- долгие или сетевые операции не блокируют GUI thread;
- callbacks из сетевого слоя доставляют изменения состояния потокобезопасно;
- для спорных мест добавлены тесты или комментарии с объяснением выбранного ограничения.

Заметка после MVP:

- текущий `ControlTrace` хранит ограниченный in-memory буфер для графика, а не полный архив испытания;
- позднее нужно решить, как сохранять полную историю управляющих воздействий и телеметрии: в один файл или в отдельные
  файлы для trace/telemetry;
- отдельно выбрать формат хранения: CSV/TSV для простого анализа, JSON/TOML для самодокументируемой структуры,
  бинарный формат или SQLite для длинных испытаний и быстрых выборок;
- это не входит в MVP, но важно не смешать краткоживущий буфер графика с долговременным журналом испытаний.
