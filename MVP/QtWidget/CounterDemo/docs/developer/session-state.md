# SessionState

`SessionState` хранит текущее in-memory состояние приложения: параметры испытания, состояние стенда, телеметрию,
модели графиков, данные PDF-протокола и настройки интерфейса.

Этот объект не является долговременным хранилищем и не является потокобезопасной базой данных. Он используется как
центральный снимок состояния MVP в пределах одного процесса.

## Поток выполнения

На текущем этапе `SessionState` рассчитан на использование только из основного Qt-потока.

Это означает:

- чтение через `get()` выполняется из основного Qt-потока;
- запись через методы `set...()` выполняется из основного Qt-потока;
- подписчики получают уведомления в том же потоке, в котором был вызван метод изменения состояния;
- сетевые callbacks и таймеры не должны напрямую менять `SessionState` из worker thread.

Метод `get()` возвращает ссылку на `SessionStateData`. Это допустимо только из-за single-thread ограничения. Если в
проект будет введена настоящая многопоточность, этот контракт нужно пересмотреть и заменить прямой доступ на snapshot
или queued-dispatch модель.

## Атомарные изменения состояния

Связанные поля должны меняться одним методом, чтобы UI не видел промежуточные неконсистентные состояния.

Пример: тип испытания, режим управления стендом, источник времени и направление времени меняются через единый переход
`setTestModeState(...)`.

Нежелательный подход:

```text
setTestProtocolMode(...)
setStandControlMode(...)
setTestTimeSource(...)
setTestTimeDirection(...)
```

Каждый такой вызов публикует отдельное уведомление. UI может получить состояние, где часть полей уже обновлена, а часть
ещё нет.

## Правило для новых use case

Если use case меняет несколько связанных полей состояния, сначала нужно проверить, не требуется ли отдельный атомарный
метод в `SessionState`.

Критерий: если UI или presenter не должны видеть промежуточные значения, изменение должно публиковаться одним
уведомлением.
