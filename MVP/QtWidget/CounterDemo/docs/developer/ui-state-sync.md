# Синхронизация UI и состояния

Пользовательский интерфейс получает состояние через цепочку:

```text
SessionState -> SessionStateQtAdapter -> QWidget
```

Команды оператора идут в обратную сторону:

```text
QWidget -> Presenter -> UseCase -> SessionState
```

Эти два направления нельзя смешивать.

## Запрещённый цикл

Опасный сценарий:

```text
State changed
-> UI программно меняет widget
-> widget испускает valueChanged/currentIndexChanged/textChanged
-> вызывается presenter/use case
-> снова меняется State
```

Такой цикл уже приводил к рекурсивному обновлению интерфейса и падению приложения.

## Правило программного обновления widgets

Если widget меняется из обработчика сигнала `SessionStateQtAdapter`, обновление должно быть защищено от обратного
сигнала.

Для следующих widget-типов по умолчанию используется `QSignalBlocker`:

- `QComboBox`;
- `QSpinBox`;
- `QDoubleSpinBox`;
- `QLineEdit`;
- `QCheckBox`;
- `QScrollBar`, если изменение значения не должно считаться действием оператора.

Исключение допустимо только если сигнал widget-а не подключён к presenter/use case или если есть другой явный guard.

## Перестроение графиков

Перестроение графика из state-change обработчика не должно синхронно запускать цепочку, которая снова меняет
`SessionState`.

Если перестроение может привести к изменению состояния, оно должно быть:

- отложено через Qt event loop;
- защищено guard-флагом от повторного планирования;
- либо перенесено в application layer так, чтобы UI только отображал уже готовую модель.

## Практическая проверка

При ревью UI-кода нужно искать:

- `setValue(...)`;
- `setText(...)`;
- `setCurrentIndex(...)`;
- `setChecked(...)`;
- `setRange(...)`, если рядом есть изменение текущего значения.

Если такой вызов находится в обработчике сигнала от `SessionStateQtAdapter`, рядом должен быть `QSignalBlocker` или
понятное объяснение, почему обратного use case не будет.
